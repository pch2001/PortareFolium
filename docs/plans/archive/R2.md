## 1. 배경 (왜 이전하는가)

PortareFolium은 현재 Supabase Storage `images` bucket을 이미지 CDN으로 사용 중이며, **Supabase free plan의 cached egress 한계(월 5 GB)에 도달**해 추가 트래픽이 차단되는 상황. 무료 한도 내에서 운영하면서 향후 확장에도 비용 폭증 위험 없는 대안 필요.

**대안 비교 결과**:

| 서비스 | Storage 무료 | Egress 무료 | 적합도 |
|---|---|---|---|
| **Cloudflare R2** ⭐ | 10 GB | **무제한 ($0)** | ⭐⭐⭐⭐⭐ |
| ImageKit | 무제한 (origin 연결) | 25 GB/월 | ⭐⭐⭐⭐ |
| Vercel Blob | 1 GB | 캡 후 과금 | ⭐⭐ (Supabase와 같은 한계) |
| bunny.net | $0.01/GB | $0.01/GB | ⭐⭐ (저렴하나 무료 X) |
| Cloudinary | 25 credits/월 | credit 차감 | ⭐ (overkill) |

**선택**: **Cloudflare R2** — egress $0이 결정적. 향후 트래픽 증가해도 비용 리스크 없음.

**Image Transformations 제외 사유**: custom domain이 없고 Vercel 무료 도메인(`gvm1229-portfolio.vercel.app`)만 사용 중. Cloudflare URL Image Transformations은 Cloudflare zone에 등록된 custom domain이 필요하므로 현재 환경에서 사용 불가. 기존 클라이언트 측 `toWebPBlob()` WebP 변환을 그대로 유지하여 이미지 최적화 처리.

## 2. Wrangler 도입 배경

참고 프로젝트 mollulog는 Cloudflare Workers에 배포되어 D1/KV/R2 바인딩을 네이티브로 사용. PortareFolium은 **Vercel + Next.js** 배포이므로 Workers 바인딩을 런타임에서 직접 사용할 수 없지만, **Wrangler CLI를 인프라 관리 도구**로 도입하여 다음을 얻음:

| 용도 | Wrangler CLI | 기존 방식 (dashboard / SDK) |
|---|---|---|
| R2 bucket 생성/삭제 | `wrangler r2 bucket create` | Cloudflare dashboard 클릭 |
| R2 public access 설정 | dashboard (CLI 미지원) | 동일 |
| 파일 업로드 (migration) | `wrangler r2 object put` | S3 SDK `PutObjectCommand` |
| 파일 조회/삭제 | `wrangler r2 object get/delete` | S3 SDK |
| 로컬 R2 에뮬레이션 | `wrangler dev` (miniflare) | 없음 (Supabase Storage mock) |
| Cloudflare 설정 관리 | `wrangler.jsonc` (IaC) | 환경변수 + dashboard |
| **Production upload (Vercel)** | **S3 SDK 필요** (Workers 바인딩 불가) | S3 SDK |

**결론**: Wrangler는 CLI 관리 + 마이그레이션 + 로컬 개발용. Production 업로드는 Vercel 런타임이므로 S3 호환 API(`@aws-sdk/client-s3`) 유지 필수.

## 3. 최종 아키텍처

```
업로드 (admin):
  client → toWebPBlob() 변환 (기존 유지)
        → POST file (WebP)
        → Vercel API route (admin auth)
        → S3 SDK → R2 PutObject

마이그레이션 / CLI 관리:
  wrangler r2 object put/get/delete
  wrangler.jsonc로 bucket 설정 관리

로컬 개발:
  wrangler dev → miniflare R2 에뮬레이션 (선택적)

저장:
  R2 bucket `gvm1229-portfolio-images`
  WebP 형식 (클라이언트에서 변환 완료된 상태)

딜리버리:
  <img src="https://pub-<hash>.r2.dev/<key>"/>
        → R2 public access (r2.dev subdomain)
        → egress $0
```

**핵심 결정**:

- **인프라 관리**: Wrangler CLI + `wrangler.jsonc` (IaC)
- **Production upload**: S3 SDK via Vercel API route (Workers 바인딩은 Vercel에서 불가)
- **WebP 변환**: 기존 client-side `toWebPBlob()` **유지** (Image Transformations 미사용)
- **Public URL**: `https://pub-<hash>.r2.dev` (custom domain 없음)
- **DB URL rewrite**: MCP tool 우선 (자동 revalidation 포함) + non-MCP 테이블만 direct UPDATE

## 5. Supabase → Cloudflare D1 마이그레이션 평가

참고 프로젝트 mollulog가 D1 + Drizzle ORM으로 운영 중이므로 동일 전환 가능성을 평가.

### 현재 구성 (Supabase PostgreSQL)

- Auth (이메일/소셜 로그인, 세션 관리)
- RLS (Row Level Security — admin/public 접근 제어)
- PostgreSQL (JSON 연산자, full-text search, 복합 인덱스)
- Realtime (현재 미사용이나 확장 가능)
- 테이블 12+ / 마이그레이션 시스템 (`auto-migrate.ts`)
- MCP 서버가 Supabase 직접 쿼리
- 모든 admin 패널이 `browserClient` (anon key + RLS) 사용

### D1 전환 시 필요한 작업

| 항목 | 비용 |
|---|---|
| Auth 시스템 재구현 | Cloudflare Access 또는 자체 구현 필요 (Supabase Auth 대체 없음) |
| RLS 재구현 | D1에 RLS 없음 → application layer에서 직접 구현 |
| 쿼리 전량 재작성 | PostgreSQL → SQLite 방언 전환 (JSON 연산자, RETURNING 제한 등) |
| ORM 도입 | Drizzle ORM 스키마 + 마이그레이션 전량 재작성 |
| MCP 서버 재작성 | `lib/mcp-tools.ts` + `lib/queries.ts` 전면 교체 |
| Admin 전체 재작성 | `browserClient` 호출 → API route 경유 D1 접근으로 변경 |
| Vercel ↔ D1 연결 | Workers 바인딩 불가 → REST API (`/d1/database/.../query`) 또는 별도 Worker proxy 필요 |

### D1 전환 이점

| 항목 | 효과 |
|---|---|
| DB egress $0 | 현재 Supabase DB egress 문제 **없음** (Storage egress만 문제) |
| SQLite 단순성 | PostgreSQL 기능(JSON, RLS, Auth)을 많이 사용 중이라 오히려 기능 후퇴 |
| Cloudflare 에코시스템 통합 | Vercel 배포를 유지하는 한 네이티브 바인딩 불가 |

### 결론: **전환 비추천**

현재 문제(Storage egress)는 R2 마이그레이션으로 해결됨. DB 자체에 egress/비용 문제 없음. D1 전환은 Auth + RLS + 쿼리 + MCP + Admin 전면 재작성이 필요하며, Vercel 배포를 유지하는 한 네이티브 바인딩 이점도 없음. 비용 대비 이득이 없으므로 **Supabase PostgreSQL 유지**.

> D1 전환이 합리적이 되려면: (1) Vercel → Cloudflare Workers 배포 전환 + (2) Supabase Auth 대체재 확보 + (3) PostgreSQL 의존 기능 제거 — 세 조건 모두 충족 시에만 검토.

## 6. 사용자 선행 작업 (Claude는 못 함)

실행 시점에 아래 단계를 먼저 수행해 주세요. 약 15분.

### Step U1: Wrangler CLI 설치 + Cloudflare 로그인

```bash
pnpm add -D wrangler
pnpm wrangler login
```

브라우저에서 Cloudflare 인증 완료. `wrangler whoami`로 확인.

### Step U2: R2 bucket 생성 (Wrangler CLI)

```bash
pnpm wrangler r2 bucket create gvm1229-portfolio-images
```

> Location hint는 CLI에서 `--location apac` 옵션 지원 여부 확인 후 적용. 미지원 시 dashboard에서 변경.

### Step U3: R2 public access 활성화 (Dashboard)

Wrangler CLI는 현재 public access 토글을 지원하지 않으므로 **dashboard에서 수행**:

1. Cloudflare dashboard → R2 → `gvm1229-portfolio-images` → Settings
2. "Public access" → r2.dev subdomain **"Allow Access"** 활성화
3. 발급된 `https://pub-<hash>.r2.dev` URL 복사

### Step U4: R2 S3 호환 API token 발급 (Vercel production용)

Wrangler CLI는 로컬에서 자체 OAuth 인증을 사용하지만, **Vercel production 환경에서는 S3 호환 API token이 필요**.

> R2 전용 토큰은 일반 Cloudflare API token (account/zone/user 퍼미션)과 **별도 페이지**에서 생성. 일반 API token 페이지에서 만들지 말 것.

1. Cloudflare dashboard → **R2 Object Storage** 페이지
2. 우측 **Account Details** 섹션 → **API Tokens** 옆 **"Manage"** 클릭
3. 토큰 유형 선택:
   - **Account API token** 권장 (계정에 귀속, 사용자 탈퇴해도 유효)
   - User API token도 가능 (개인 사용자에 귀속)
4. Permission: **Object Read & Write**
5. Bucket scope: **Specify bucket(s)** → `gvm1229-portfolio-images` 선택 (Object Read/Write 선택 시 bucket 범위 지정 가능)
6. TTL: 기본 (만료 없음)
7. **"Create API Token"** 클릭
8. **발급 직후만 표시되는 값 2개 즉시 복사 저장**:
   - `Access Key ID`
   - `Secret Access Key` (이후 재조회 불가)
9. S3 endpoint URL은 토큰 발급 화면이 아닌 R2 개요 페이지에서 확인:
   - 형식: `https://<account-id>.r2.cloudflarestorage.com`
   - Account ID도 같은 페이지에서 확인 가능

### Step U5: Vercel 환경변수 등록

Vercel project → Settings → Environment Variables (Production + Preview + Development 모두):

| 변수 | 값 |
|---|---|
| `R2_ACCOUNT_ID` | Step U4의 Account ID |
| `R2_ACCESS_KEY_ID` | Step U4의 Access Key ID |
| `R2_SECRET_ACCESS_KEY` | Step U4의 Secret Access Key |
| `R2_BUCKET` | `gvm1229-portfolio-images` |
| `R2_PUBLIC_URL` | Step U3에서 복사한 `https://pub-<hash>.r2.dev` URL |

`.env.local`에도 동일하게 추가 (로컬 개발용).

### Step U6: DB snapshot 백업 (안전망)

admin → Snapshots 패널 → "현재 상태 스냅샷 생성" 1회 클릭. URL rewrite 작업 중 사고 시 복구 기준점.

### Step U7: Claude에 "ready" 신호

위 단계 완료 후 Discord에 "ready" + `R2_PUBLIC_URL` 값 + 결정 사항 (c) MCP endpoint 확인 전달.

### Step C3: 마이그레이션 스크립트 — DB URL rewrite

- `scripts/rewrite-image-urls.ts` 신규
- **MCP 사용 (posts, portfolio_items)**:
  1. `list_posts` → 전체 slug
  2. 각 slug에 `get_post` → content + thumbnail + og_image
  3. JS regex로 `https://<supabase-project>.supabase.co/storage/v1/object/public/images/` → `${R2_PUBLIC_URL}/` 치환
     - markdown: `![alt](url)`
     - HTML: `<img src="url" .../>`
     - 기타 임의 attribute (background-image url 등) — base URL 단일 치환으로 자동 커버
  4. `update_post`로 변경 fields 전달 → 자동 ISR revalidation
  5. 동일 패턴으로 `list_portfolio_items` → `get_portfolio_item` → `update_portfolio_item`
- **Direct DB UPDATE (MCP tool 없는 테이블)**:
  - `about_data.profile_image` 및 JSON 필드
  - `resume_data` JSON 내 image 필드 (또는 MCP `update_resume` 사용)
  - `site_config` JSON 내 (logo, og_image 등)
  - `books.thumbnail`, `books.content`
  - 변경 후 admin Server Action `revalidatePath('/about')`, `/resume`, `/`, `/books/[slug]` 호출
- **dry-run flag 필수** — 첫 실행은 변경될 URL 목록만 출력, 사용자 확인 후 `--apply` 재실행

| Vercel function payload 4.5MB 초과 | 큰 이미지 업로드 실패 | 클라이언트에서 WebP 변환 후 전송하므로 대부분 4.5MB 이하. 초과 시 사전 압축 안내. 추후 presigned URL로 전환 가능 (이번 plan에 path 열어둠). |
| MCP endpoint 변경 | URL rewrite 실패 | Step C3 시작 전 endpoint 확인 (사용자에게 결정사항 (c) 재확인). |
| Wrangler CLI 버전 호환 | 명령어 변경/삭제 | `wrangler.jsonc`에 compatibility_date 명시. wrangler 메이저 업데이트 시 scripts 확인. |
| 30일 cleanup 전 Supabase 삭제 | 롤백 불가 | DB snapshot (Step U6) + 30일 유예 + Supabase Storage는 cleanup 명시적 사용자 승인 후에만. |

## 9. Acceptance Criteria

1. `wrangler.jsonc`에 R2 bucket binding 정의됨.
2. R2 bucket에 기존 Supabase Storage의 모든 파일이 동일 path로 복사됨 (40개 검증).
3. DB의 모든 이미지 URL 컬럼 / JSON 필드가 R2 URL로 변경됨 (전수 schema 조사 결과 기준).
4. admin 페이지에서 새 이미지 업로드 → R2에 저장됨 (Vercel function 경유).
5. 모든 frontend 페이지(/, /about, /resume, /portfolio, /portfolio/[slug], /blog, /blog/[slug], /books/[slug])에서 이미지가 R2 URL(`pub-<hash>.r2.dev`)로 정상 로딩됨.
6. `pnpm test` + `pnpm test:e2e` 통과 (회귀 없음).
7. 30일 검증 후 Supabase Storage `images` bucket 삭제 (또는 사용자 명시 시점).
8. AGENTS.md / README / CHANGES.md 갱신 (Wrangler 포함).

## 10. 나중 고려 (out of scope, 메모용)

- **Presigned URL 전환**: 단일 이미지 4.5MB+ 또는 사용자 업로드 기능 추가 시. `/api/upload-image/sign` endpoint + 클라이언트 PUT + R2 CORS 1회 설정. 약 30% 추가 코드.
- **Cloudflare Worker upload proxy**: Vercel API route 대신 CF Worker를 R2 업로드 프록시로 사용. Workers는 R2 네이티브 바인딩 가능 → S3 SDK 제거 가능. 단, 배포 대상이 2곳(Vercel + CF Workers)으로 증가.
- **Cloudflare Image Transformations**: custom domain 구매 시 도입 가능. `pub-<hash>.r2.dev` → `cdn.<domain>` 전환 + `/cdn-cgi/image/...` URL wrapping 추가. 클라이언트 `toWebPBlob()` 제거하고 delivery 시점 자동 최적화(format=auto AVIF/WebP, resize)로 대체 가능.
- **Cloudflare D1 마이그레이션**: §5 평가 결과 현재 비추천. 조건 변경 시 (Vercel → Workers 전환 등) 재평가.
- **`data-testid="image"` 부여**: E2E에서 이미지 element 안정적으로 잡기 위해. 현재는 placeholder 기반.
- **이미지 hash 기반 중복 검출**: `image-upload.ts`에 TODO로 남아 있음 (현재 plan에선 보류 유지).
- **Multi-region R2**: 글로벌 사용자 트래픽 증가 시 jurisdiction 변경. 현재 단일 region (APAC) 충분.
